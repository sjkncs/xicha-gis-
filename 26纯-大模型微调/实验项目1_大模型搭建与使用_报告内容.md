### 三、实验方案与步骤

#### 3.1 基础理论

##### 3.1.1 大模型训练的基本概念

**语言模型（LM）** 的本质是根据已知的前文内容，预测下一个最可能出现的 token（词元）。这个过程在训练时通过最大化似然概率来完成，在推理时则通过自回归采样逐步生成文本。MiniMind 项目采用 Transformer 架构作为基础，这一架构由 Vaswani 等人在 2017 年提出，核心是自注意力（Self-Attention）机制，能够高效建模序列中任意两个位置之间的依赖关系。

**MiniMind 项目** 是一个从零实现大模型训练链路的开源项目。其最小版本仅需 25.8M 参数，体积仅为 GPT-3 的 1/7000，适合在普通个人 GPU 上快速训练和运行。项目包含完整的预训练（Pretrain）和有监督微调（Supervised Fine-Tuning, SFT）两个阶段，能够帮助开发者以极低成本理解大模型训练的全流程。

##### 3.1.2 GPU 与 CUDA 环境

深度学习模型的训练需要大量的矩阵运算。GPU（图形处理器）相比 CPU 拥有成百上千个小型计算核心，能够并行处理大量矩阵乘法运算，因此成为大模型训练的硬件基础。NVIDIA GPU 通过 CUDA（Compute Unified Device Architecture）架构向开发者提供通用计算能力。

**nvidia-smi** 是 NVIDIA 提供的系统管理界面工具，用于查看显卡型号、驱动版本、显存使用情况和驱动支持的最高 CUDA 版本等关键信息。驱动支持的 CUDA 版本（如 12.x）代表硬件能力上限，而实际使用 PyTorch 时，还需要确保 PyTorch 编译时绑定的 CUDA 运行时版本与驱动兼容。

**torch.cuda.is_available()** 是 PyTorch 提供的接口，用于判断当前环境中 PyTorch 是否能够成功调用 NVIDIA GPU。返回 True 表示可以使用 GPU 加速训练，返回 False 则意味着训练将退化为 CPU 模式，速度会显著降低。

##### 3.1.3 Tokenizer 的作用

**文本不能直接输入神经网络。** 神经网络处理的输入是数值向量（Tensor），而非人类可读的文本字符。Tokenizer（分词器）承担了将文本转换为模型可处理的数字序列的关键任务，其工作流程如下：

| 步骤 | 内容 | 示例 |
|------|------|------|
| 原始文本 | 人能直接读懂的文字 | "为什么天空是蓝色的？" |
| Token 切分 | 按词表规则将文本切分为 token | "为什么 / 天空 / 是 / 蓝色 / 的 / ？" |
| ID 映射 | 每个 token 映射到词表中的整数编号 | 101, 234, 567, 890, 432, 115 |
| Embedding | 将整数 ID 转换为稠密向量矩阵 | 模型用向量进行计算 |

本实验中，同一套 tokenizer 应贯穿预训练、微调和推理全过程，以保证输入输出编号体系的一致性。

##### 3.1.4 预训练（Pretraining）

预训练的目标是让模型学习语言的基本规律和通用知识。在预训练阶段，模型在大规模纯文本语料上进行**自回归语言建模**——给定前文，预测下一个 token。预训练不提供标注信号（无监督学习），模型通过大量阅读文本自动总结语言的统计规律。

预训练完成后，模型具备强大的文本续写能力，但尚不具备明确的"助手"身份意识，回答风格偏向自由生成而非结构化问答。

预训练阶段的训练效果通过 **loss（损失值）** 来衡量。Loss 越低，表示模型对训练数据的拟合程度越好。观察 loss 曲线是否整体下降、是否出现 NaN 异常，是判断预训练是否正常进行的重要依据。

##### 3.1.5 SFT 微调（Supervised Fine-Tuning）

SFT（Supervised Fine-Tuning，有监督微调）的目标是将预训练阶段学到的语言能力，调整为更适合对话和任务执行的输出方式。与预训练使用纯文本不同，SFT 使用**多轮对话格式**的数据，每条数据包含 user（用户）和 assistant（助手）的交替对话。

SFT 的核心意义在于：
1. **指令跟随**：让模型学会理解用户指令并给出符合要求的回答
2. **身份对齐**：建立统一的助手口径，避免身份回答的不一致
3. **格式规范**：使回答更结构化、更易于理解

SFT 不是简单"让模型更聪明"，而是将预训练已有的语言能力塑造成符合人类偏好的输出形式。

##### 3.1.6 预训练与 SFT 的对比

| 对比维度 | 预训练模型 | SFT 微调后模型 |
|----------|-----------|----------------|
| 回答形式 | 更像文本续写或自由生成 | 更像问答助手 |
| 身份口径 | 回答口径可能不稳定 | 更容易形成统一助手口径 |
| 任务遵循 | 可能偏题或格式混乱 | 更能按问题要求组织回答 |
| 语言流畅度 | 可能已有基础表达能力 | 表达通常更清晰、有条理 |
| 应用场景 | 文本补全、写作辅助 | 智能问答、任务执行 |

##### 第一步：环境检查

**目的**：确认本地或服务器是否具备 GPU 训练条件。

**操作步骤**：

1. 打开 Anaconda Prompt 终端
2. 执行 `nvidia-smi` 命令，检查显卡型号、驱动版本和 CUDA 版本
   - 重点关注：GPU Name（显卡型号）、Driver Version（驱动版本）、CUDA Version（驱动支持的最高 CUDA 版本）、Memory-Usage（显存使用情况）
3. 如电脑有多个显卡，执行 `wmic path win32_VideoController get name` 查看 Windows 识别到的所有显卡
4. 执行 PyTorch CUDA 检查命令验证 PyTorch 是否能调用 GPU：

```bash
python -c "import torch; print('torch 版本:', torch.__version__); print('CUDA 是否可用:', torch.cuda.is_available()); print('torch 使用的 CUDA 版本:', torch.version.cuda); print('显卡名称:', torch.cuda.get_device_name(0) if torch.cuda.is_available() else 'No CUDA')"
```

**判断标准**：
- `torch.cuda.is_available()` 返回 **True** → PyTorch 已能调用 NVIDIA GPU，可进行 GPU 加速训练
- `torch.cuda.is_available()` 返回 **False** → 训练将退化为 CPU 模式，应检查驱动、PyTorch CUDA 版本和环境是否激活

##### 第二步：环境配置

**目的**：创建独立的 Python 环境，避免包版本冲突。

**操作步骤**：

1. 确认 conda 已安装：
```bash
conda --version
```

2. 创建名为 minimind 的 Python 3.10 环境：
```bash
conda create -n minimind python=3.10 -y
```

3. 激活环境：
```bash
conda activate minimind
```
激活后命令行前应出现 `(minimind)` 标识。每次重新打开终端都需要重新激活环境。

4. 升级 pip：
```bash
python -m pip install --upgrade pip
```

5. 如需安装 PyTorch（已预装则跳过），前往 [PyTorch 官网](https://pytorch.org/get-started/locally/)，选择 Windows + Pip + 对应 CUDA 版本，执行安装命令。

##### 第三步：项目下载与依赖安装

**目的**：获取 MiniMind 项目代码和所需依赖。

**操作步骤**：

1. 创建项目目录（如尚未创建）：
```bash
mkdir D:\AI
cd /d D:\AI
```

2. 克隆 MiniMind 仓库：
```bash
git clone --depth 1 https://github.com/jingyaogong/minimind
```

3. 进入项目目录并安装依赖：
```bash
cd minimind
pip install -r requirements.txt -i https://mirrors.aliyun.com/pypi/simple
```

##### 第四步：数据准备

**目的**：下载训练数据并观察预训练数据与 SFT 数据的格式差异。

**操作步骤**：

1. 创建数据集存放目录：
```bash
mkdir dataset
```

2. 通过 ModelScope 下载 mini 数据集：
```bash
pip install -U modelscope
modelscope download --dataset gongjy/minimind_dataset --include pretrain_t2t_mini.jsonl sft_t2t_mini.jsonl --local_dir dataset
```

3. **观察数据格式差异**：
   - `pretrain_t2t_mini.jsonl`：纯文本语料，每行是一个完整的文本片段，用于预训练阶段学习语言规律
   - `sft_t2t_mini.jsonl`：多轮对话格式，每行包含 `conversations` 字段，其中每个对象有 `role`（user/assistant）和 `content`（对话内容），用于 SFT 阶段学习指令跟随和助手式回答

##### 第五步：预训练

**目的**：使用纯文本语料训练模型，使其学习语言的基本规律。

**操作步骤**：

1. 进入项目根目录，在地址栏输入 `cmd` 回车打开终端
2. 激活 minimind 环境：
```bash
conda activate minimind
```
3. 进入 trainer 目录：
```bash
cd trainer
```
4. 启动预训练脚本：
```bash
python train_pretrain.py
```
5. 观察训练输出：
   - Loss 值应整体呈下降趋势
   - 每 1000 批次自动保存一次模型权重到 `out/pretrain_768.pth`
   - 可通过 SwanLab 观察 loss 曲线
6. 如需中断训练，按 `Ctrl+C`（如卡住可多按几次）

##### 第六步：SFT 微调

**目的**：使用对话格式数据微调预训练模型，使模型学会按助手风格回答问题。

**前置条件**：确保 `out/` 文件夹中已有 `pretrain_768.pth` 预训练权重文件。

**操作步骤**：

1. 确保在 minimind 环境
2. 进入 trainer 目录：
```bash
cd trainer
```
3. 启动 SFT 微调脚本：
```bash
python train_full_sft.py
```
4. 观察训练输出：Loss 值应稳定下降
5. 每 1000 批次自动保存权重到 `out/full_sft_768.pth`

> **注意**：由于完整训练时间较长，可提前中断训练（Ctrl+C），只需保证各步骤代码运行通畅并理解其应用即可。模型评估可使用已训练好的模型权重。

##### 第七步：模型评估与对比

**目的**：测试预训练模型和 SFT 微调模型的实际回答效果，对比两者的差异。

**操作步骤**：

1. 将已训练模型权重放入 `out/` 目录
2. 在项目根目录打开终端，激活 minimind 环境
3. 测试预训练模型：
```bash
python eval_llm.py --weight pretrain
```
4. 测试 SFT 微调模型：
```bash
python eval_llm.py --weight full_sft
```

**可用模型权重说明**：

| 权重名称 | 说明 |
|----------|------|
| `pretrain_1_1000` | 第 1 个 epoch 中 1000 批次训练的模型（1 epoch = 39695 批次） |
| `pretrain_2_1000` | 第 2 个 epoch 中 1000 批次训练的模型 |
| `pretrain1` | 2 个 epoch 完整训练的模型 |
| `full_sft_1_1000` | SFT 第 1 个 epoch 中 1000 批次训练的模型（1 epoch = 56608 批次） |
| `full_sft_2_1000` | SFT 第 2 个 epoch 中 1000 批次训练的模型 |
| `full_sft1` | SFT 2 个 epoch 完整训练的模型 |

**效果对比维度**（测试问题："具身智能是什么"）：

| 对比维度 | 预训练模型 | SFT 微调模型 |
|----------|-----------|----------------|
| 回答形式 | 自由续写，无明确定义框架 | 以"具身智能是…"开篇，结构清晰 |
| 内容组织 | 从任意角度展开 | 先定义概念，分点解释，举例说明 |

### 四、实验结果与分析

#### 4.1 环境检查结果

本环节通过一系列系统命令完成对本机硬件与软件环境的全面检查。

**nvidia-smi 输出分析**：

执行 `nvidia-smi` 命令后，输出包含以下关键信息：
- **GPU Name**：显示本机所装备的 NVIDIA 显卡具体型号（如 RTX 3060、RTX 4090 等）
- **Driver Version**：NVIDIA 驱动版本号，驱动版本直接影响可支持的 CUDA 版本上限
- **CUDA Version**：驱动支持的最高 CUDA 运行版本（如 12.x），此为驱动层面的能力上限，不代表已安装完整 CUDA Toolkit
- **Memory-Usage**：当前显存占用情况，训练大模型时显存不足会触发 "CUDA out of memory" 错误

若执行 `nvidia-smi` 报错，通常原因包括：本机无 NVIDIA 显卡、驱动未安装、驱动安装后未重启系统、命令行路径未生效等。

**PyTorch CUDA 检查结果**：

执行 Python 检查命令后，应关注以下输出项：
- `torch.__version__`：PyTorch 版本号，需与 Python 3.10 环境兼容
- `torch.cuda.is_available()`：**核心判断指标**，True 表示 GPU 可用，False 表示 CPU 模式
- `torch.version.cuda`：PyTorch 绑定的 CUDA 运行时版本，应与驱动版本兼容
- `torch.cuda.get_device_name(0)`：检测到的第一块 NVIDIA 显卡名称

#### 4.2 数据格式观察结果

**预训练数据（pretrain_t2t_mini.jsonl）格式**：

```json
{"text": "这是一段纯文本内容，用于预训练阶段学习语言规律。"}
{"text": "另一段独立的纯文本语料，每行对应一个完整的文本片段。"}
```

特点：每行是一个 JSON 对象，包含单一的 `"text"` 字段，值为未经结构化的纯文本。模型在预训练中学习的是"给定前文，预测下一个词"的语言建模能力，不涉及角色或对话格式。

**SFT 数据（sft_t2t_mini.jsonl）格式**：

```json
{"conversations": [
  {"role": "user", "content": "你是什么模型？"},
  {"role": "assistant", "content": "我是由 jingyaogong 开发的高效小参数 AI 模型。"}
]}
```

特点：每行是一个 JSON 对象，包含 `"conversations"` 数组字段，数组内以 `role`（user/assistant/system）和 `content` 字段的交替形式组织多轮对话。模型在 SFT 中学习的是"理解用户指令 → 给出符合助手身份的规范化回答"的映射关系。

**两种数据的核心区别**：

| 维度 | 预训练数据 | SFT 数据 |
|------|-----------|----------|
| 结构 | 单一 text 字段 | 多轮对话数组 |
| 内容 | 纯文本自由生成 | 指令-回答配对 |
| 作用 | 学习语言规律 | 学习指令跟随 |
| 格式 | 无格式约束 | 有角色和格式约束 |

#### 4.3 预训练结果分析

预训练阶段的核心观察指标是 **loss（损失值）** 随训练步数的变化。

**loss 曲线分析**：
- 正常情况下，loss 应整体呈**下降趋势**，表明模型对训练数据的拟合能力在逐步提升
- 若 loss 出现 **NaN**（Not a Number），通常意味着梯度爆炸或学习率设置过高，需立即中断训练并调整超参数
- 若 loss 下降过快后趋于平坦，说明模型可能已接近该数据规模下的收敛状态

**训练稳定性观察**：
- 每隔 1000 批次自动保存一次权重文件 `pretrain_768.pth`，可通过 SwanLab 实验平台实时观察 loss 曲线
- 训练过程中显存占用应保持稳定，若显存持续增长则可能存在内存泄漏问题

**预训练模型能力评估（eval_llm.py）**：

使用测试问题**"具身智能是什么"**对预训练模型进行推理时，观察到以下典型行为：
- 模型倾向于**文本续写**而非结构化回答——输入问题后，模型可能直接围绕"具身"或"智能"相关词汇续写下去，而非以"具身智能是…"的定义式回答开篇
- 对于概念解释类问题，预训练模型可能会从某个角度（如哲学、机器人学）展开自由续写，**缺乏对问题类型的明确判断**，无法自动形成"先定义、后解释、再举例"的规范结构
- 回答内容可能包含一定相关性但**不够准确或完整**，因为预训练阶段学到的是语言的统计规律而非精确的知识图谱
- 整体输出偏向于**语言流畅但结构松散**的形式

#### 4.4 SFT 微调结果分析

SFT 微调阶段在预训练权重的基础上，使用对话数据进一步训练模型。

**微调训练观察**：
- Loss 值同样应呈下降趋势，与预训练类似，但收敛速度可能因数据量更小而更快
- 微调阶段使用 `train_full_sft.py` 脚本启动，权重每 1000 批次保存为 `full_sft_768.pth`
- 与预训练相同，完整训练时间较长，实验时可提前中断，只需确保代码运行通畅

**SFT 模型能力评估（eval_llm.py）**：

使用相同测试问题对 SFT 微调模型进行推理时，观察到以下改善：

| 对比维度 | 预训练模型表现 | SFT 微调后表现 |
|----------|--------------|----------------|
| 回答形式 | 自由续写，无明确边界 | 以"我是…"开头，结构清晰 |
| 身份口径 | 不稳定，可能自称为各类模型 | 统一口径，声称由 jingyaogong 开发 |
| 任务遵循 | 可能答非所问或格式混乱 | 理解问题，按要求组织回答 |
| 对话格式 | 无 user/assistant 区分意识 | 明确以助手身份回应 |
| 语言组织 | 基本通顺但较松散 | 表达清晰，有条理 |

#### 4.5 预训练与 SFT 效果对比总结

以测试问题**"具身智能是什么"**为例，通过实验观察到以下差异：

| 模型类型 | 典型回答特征 |
|---------|------------|
| 预训练模型 | 围绕"具身"或"智能"关键词自由续写，缺乏明确的定义框架，内容相关性存在但结构松散 |
| SFT 微调模型 | 以"具身智能是…"开篇形成定义框架，分点解释概念含义，举例说明应用场景，格式规范有边界 |

**核心结论**：**预训练解决的是"会说人话"的问题，SFT 解决的是"按人类期望的方式说"的问题**。两者功能互补，共同构成完整的大模型训练链路。

### 五、思考题回答

**思考题 1：为什么原始文本不能直接送入神经网络，而要先经过 Tokenizer？**

神经网络本质上是数值计算系统，其输入必须是浮点数或整数的张量（Tensor），而人类可读的文本字符无法被直接处理。Tokenizer（分词器）的核心作用是将离散的文本符号转换为连续的数值表示，具体包括三个步骤：

第一，**分词（Tokenization）**：按照预定义的词表规则，将输入文本切分为 token 序列。不同语言的 tokenization 策略有所不同，中文可按字/词切分，英文可按 subword（子词）切分，如 BPE（Byte Pair Encoding）或 WordPiece 算法。MiniMind 项目使用的 tokenizer 即通过类似机制将文本映射为 token 列表。

第二，**ID 映射（Token-to-Index）**：将切分出的每个 token 映射为词表（Vocabulary）中的唯一整数编号。例如"天"→101，"空"→234，"蓝"→567。这样文本就变成了 [101, 234, 567, …] 的整数序列。

第三，**向量嵌入（Embedding）**：将整数 ID 通过嵌入矩阵（Embedding Matrix）查表操作，转换为 d 维稠密向量。嵌入层本身是可学习的参数，在训练过程中自动优化，使每个 token 的向量表示能够捕获语义信息。

如果不经过 tokenizer 直接输入文本字符的 Unicode 编码，则无法利用预训练学到的语义表示，且输入维度极高（每个汉字一个编码），无法有效建模语言规律。

---

**思考题 2：pretrain_t2t_mini.jsonl 和 sft_t2t_mini.jsonl 的数据格式有什么不同？**

两者最本质的区别在于数据结构和训练目标不同：

`pretrain_t2t_mini.jsonl` 采用**纯文本格式**，每行是一个独立的 JSON 对象，包含一个 `text` 字段，值为未经任何结构化处理的原始文本段落。这种格式的数据用于训练模型的"语言建模"能力——给定前文预测下一个 token。模型学到的是语言的统计规律（如词语搭配、句法结构、世界知识片段），不涉及任何角色或任务的概念。

`sft_t2t_mini.jsonl` 采用**多轮对话格式**，每行是一个 JSON 对象，包含一个 `conversations` 数组。数组内交替排列着 user（用户）和 assistant（助手）的对话片段，每条消息有 `role` 字段指明角色、`content` 字段指明内容。这种格式的数据用于训练模型的"指令跟随"能力——模型学习的是"在 user 说完之后，以 assistant 的身份给出符合预期的回答"的映射关系。

简而言之，**pretrain 数据是"给模型看书让它自己学说话"，sft 数据是"给模型看问答对话让它学会当助手"**。

---

**思考题 3：为什么预训练后的模型已经能说话，但还不一定像一个合格助手？**

预训练模型能说话，是因为它在大规模纯文本上完成了语言建模任务——通过"给定前文预测下一个 token"的训练目标，模型自动学会了语言的统计规律，包括词汇搭配、句法结构、事实性知识等。这些能力使得模型能够生成语法正确、语义连贯的文本。

但"像一个合格助手"不仅仅是"能说话"，还包括以下关键能力：

**第一，身份一致性**。预训练语料中包含各种来源、各种角色的文本，模型学到的是不同语气、不同身份混在一起的统计分布。因此预训练模型对于"你是什么模型？"这类身份问题的回答可能是任意的、不一致的——有时说自己是某开源模型，有时说自己是最先进的 AI，有时则完全答非所问。

**第二，指令理解与遵循**。助手的核心价值在于"听懂了人类的需求并完成它"。预训练阶段没有针对指令跟随进行训练，模型习惯的是"续写文本"而非"执行任务"。面对"请用三句话解释量子力学"这样的结构化请求，预训练模型可能直接围绕"量子力学"展开自由写作，而非按要求的格式和长度回答。

**第三，输出格式规范**。助手式回答通常要求清晰、有条理、有边界。预训练模型倾向于无限续写，输出边界不清晰，可能在一个回答后继续生成无关内容。

因此，预训练解决的是"语言能力"的问题，SFT 解决的是"按人类期望的方式使用语言能力"的问题。

---

**思考题 4：SFT 微调主要改善了模型的哪些表现？请结合你的推理截图说明。**

SFT 微调在以下维度对模型表现有显著改善。以测试问题**"具身智能是什么"**为例：

**（1）回答结构的规范化**。经过 SFT 后，模型面对"是什么"类问题时，会主动形成"定义→解释→举例"的结构化回答模式。例如：
- 预训练模型可能回答：*"具身智能…在认知科学中常被讨论…不同研究者对此有不同理解…"*（自由续写，无边界）
- SFT 模型则回答：*"具身智能是一种强调智能行为必须与身体经验紧密结合的研究范式…"*（有明确定义框架）

**（2）任务类型的理解与遵循**。SFT 训练数据中包含大量"是什么""怎么做""为什么"等不同类型的问题，模型学会了针对不同问题类型采用相应的回答策略，而非笼统地自由续写。

**（3）内容组织的条理性**。SFT 后模型在解释"具身智能"时，更倾向于分点论述（如"第一，具身智能强调身体的作用；第二…"），而非一段式无序展开。

**（4）对话格式的规范性**。SFT 训练中模型学会了对用户保持友好、专业的语气，输出有明确的开始和结束边界，不会无限续写。

**（5）身份一致性**。在多轮对话中，SFT 模型能始终以助手身份回应，不会出现语气突变或角色混淆的情况。

**思考题 5：如果 torch.cuda.is_available() 是 False，可能有哪些原因？你会如何排查？**

`torch.cuda.is_available()` 返回 False，意味着 PyTorch 环境中无法识别 NVIDIA GPU。常见原因及排查方法如下：

**原因一：PyTorch 安装成了 CPU 版本**

最常见的原因是使用 `pip install torch` 安装了不带 CUDA 支持的 CPU-only 版本。**排查方法**：执行 `pip show torch`，查看安装的 torch 版本是否包含 "cu" 标记（如 `cu126` 表示 CUDA 12.6 版本）。若不包含 CUDA 标记，则需要卸载后重新安装 GPU 版本：

```bash
pip uninstall torch torchvision torchaudio
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu126
```

**原因二：NVIDIA 驱动未安装或版本过旧**

即使 PyTorch 是 CUDA 版本，若本机无 NVIDIA 显卡或驱动缺失，torch.cuda.is_available() 仍会返回 False。**排查方法**：执行 `nvidia-smi` 命令，若报错"No NVIDIA driver installed"，则需要前往 NVIDIA 官网下载并安装对应显卡型号的驱动程序。

**原因三：驱动版本与 PyTorch CUDA 版本不兼容**

驱动支持的 CUDA 版本需要 ≥ PyTorch 编译时绑定的 CUDA 版本。例如驱动支持 CUDA 11.8，但 PyTorch 绑定的是 CUDA 12.6，则可能不兼容。**排查方法**：对比 `nvidia-smi` 显示的 Driver CUDA Version 和 `torch.version.cuda` 的值，前者应大于等于后者。

**原因四：当前 conda 环境未激活**

在 minimind 环境之外的系统 Python 或其他 conda 环境中，PyTorch 可能未安装或不包含 CUDA 支持。**排查方法**：确认命令行前方有 `(minimind)` 标识，执行 `conda activate minimind` 切换到正确环境后再运行检查命令。

**原因五：多显卡环境下的 GPU 可见性问题**

极少数情况下，NVIDIA 驱动检测正常但 CUDA 可见性受限。**排查方法**：在 Python 中执行 `torch.cuda.device_count()` 查看检测到的 GPU 数量是否为 0，若是 0 则需检查驱动安装或考虑重新安装 PyTorch。

**综合排查流程**：

```
1. 执行 nvidia-smi → 确认显卡和驱动是否正常
2. 确认 (minimind) 环境已激活
3. pip show torch → 确认 torch 版本包含 CUDA
4. 对比驱动 CUDA 版本 vs torch.version.cuda
5. 必要时卸载 CPU 版 PyTorch，重新安装 CUDA 版本
```
